# PLB-N32 存储策略

本文定义 PLB-N32 的 Flash 存储分层、当前落地方案和后续 OTA / bootloader / 工厂数据扩展方向。

目标：

- 区分出厂只读数据、运行期配置、高频小状态、事件日志和固件镜像。
- 避免业务层直接裸写 Flash。
- 为后续 OTA、bootloader、工厂 TLV 数据预留清晰边界。
- 保持当前 FEE / EEPROM API 可用，不破坏已落地功能。

## 当前 Flash 条件

PLB-N32 当前工程按 N32L406 片内 Flash 规划：

```text
Flash base: 0x08000000
Flash size: 0x00020000  128 KiB
Page size:  0x00000800  2 KiB
Page count: 64
```

当前已落地的末尾分区：

```text
0x0801E000 - 0x0801EFFF  EEPROM  4 KiB  page 60-61
0x0801F000 - 0x0801FFFF  FEE     4 KiB  page 62-63
```

## 存储分层

推荐存储栈：

```text
业务层
  |-- Factory TLV: 出厂身份、校准、硬件版本，只读或极少写
  |-- FEE:         运行期可更新 key-value 配置
  |-- EEPROM:      高频小定长计数器/状态字
  |-- Evtlog:      固定记录事件日志
  |-- OTA slots:   固件镜像和升级元数据
  `-- Bootloader:  启动、校验、切换和恢复

底层
  `-- Flash ops:   read / write / erase_page，只作为驱动能力
```

裸 Flash 操作只应该被上层存储组件或 bootloader 使用，不直接给普通业务保存参数。

## 当前分区职责

### EEPROM 分区

用途：小定长、高频、允许反复更新的状态。

当前地址：

```text
Base: 0x0801E000
Size: 0x1000
Page: 60-61
```

当前已使用：

```text
index 0: boot_count, uint32_t
```

适合继续放：

- boot_count
- reset/error 计数器
- 最近一次 distress 状态字
- 小 flags
- 小型单值计数器

不适合放：

- 变长字符串
- 服务器地址结构体
- 大块 blob
- 事件流水
- 固件升级元数据

原因：EEPROM 组件有 RAM shadow，读取快，写入按固定 record 追加，适合小变量。

### FEE 分区

用途：运行期可更新 key-value 配置。

当前地址：

```text
Base: 0x0801F000
Size: 0x1000
Page: 62-63
```

当前已使用：

```text
key 0x00020001: server endpoint
```

适合继续放：

- 服务器 IP / port
- MQTT broker / topic / APN 等低频配置
- 上报间隔
- 工作模式
- 可现场修改的阈值
- 最近一次有效 GNSS 位置，前提是低频写入

不适合放：

- 出厂序列号
- MMSI 原始值
- RF/GNSS 校准表
- 高频 boot_count
- 持续事件日志

原因：FEE 支持追加写、GC、页状态恢复，适合低频可变配置，但不应该替代工厂数据区。

### 裸 Flash ops

用途：底层读写擦能力。

当前 PLB-N32 已通过 `plb_n32_flash_fee.c` 封装给 FEE / EEPROM 使用。

允许使用场景：

- FEE backend
- EEPROM backend
- 后续 Factory TLV backend
- 后续 bootloader / OTA 写镜像
- 工厂治具写入整页或整分区
- Flash 自检

不建议使用场景：

- 业务层直接保存服务器地址
- 业务层直接保存 boot_count
- 业务层直接改 factory 字段

裸写没有自带 CRC、版本、事务边界和兼容迁移，业务直接使用风险高。

## 未来 Factory TLV 分区

Factory TLV 用于出厂一次性写入或极少写入的数据。它和 FEE 的定位不同：Factory TLV 是设备身份和校准的原始来源；FEE 是现场运行期覆盖配置。

建议数据：

```text
type 0x0001: serial_number
type 0x0002: mmsi
type 0x0003: hardware_revision
type 0x0004: production_date
type 0x0005: factory_test_flags
type 0x0006: rf_calibration
type 0x0007: gnss_calibration
type 0x0008: device_key_hash
type 0x0009: default_server_endpoint
```

建议格式：

```c
typedef struct {
    uint32_t magic;      /* 'FTLV' */
    uint16_t version;
    uint16_t header_len;
    uint32_t total_len;
    uint32_t seq;
    uint32_t crc32;     /* cover TLV payload */
    uint32_t flags;
} plb_factory_tlv_header_t;
```

payload 使用现有 `components/tlv`：

```text
uint16_t type
uint16_t length
uint8_t  value[length]
```

最低要求：

- magic
- version
- total_len
- crc32
- TLV iterator
- unknown type 可跳过

如果 factory 数据只在工厂烧录，一份 Factory TLV + CRC 即可。

如果后续允许现场或售后更新 factory 数据，建议双 slot：

```text
Factory A
Factory B
```

启动时选择：

```text
VALID && CRC OK && seq 最大
```

### 推荐地址预留

当前 EEPROM/FEE 已占用 page 60-63。如果 Flash 空间允许，建议在其前方预留 factory：

```text
0x0801D800 - 0x0801DFFF  Factory TLV  2 KiB  page 59
0x0801E000 - 0x0801EFFF  EEPROM       4 KiB  page 60-61
0x0801F000 - 0x0801FFFF  FEE          4 KiB  page 62-63
```

如果 factory 数据较多，使用 2 页：

```text
0x0801D000 - 0x0801DFFF  Factory TLV  4 KiB  page 58-59
0x0801E000 - 0x0801EFFF  EEPROM       4 KiB  page 60-61
0x0801F000 - 0x0801FFFF  FEE          4 KiB  page 62-63
```

最终地址必须和 linker script 协调，确保应用固件不会覆盖 factory / EEPROM / FEE 区。

## 未来 Bootloader / OTA 策略

128 KiB Flash 对 OTA 双镜像较紧张，需要先明确固件体积目标。

当前 PLB-N32 构建体积约：

```text
text + data ~= 43 KiB
```

后续加入通信栈、AT、GNSS、协议、OTA 后体积会增长。OTA 策略需要按最终固件大小选择。

### 方案 A：片内双槽 OTA

适用条件：应用固件能长期控制在较小体积，例如每槽 48 KiB 左右。

示例规划：

```text
0x08000000 - 0x08003FFF  Bootloader       16 KiB  page 0-7
0x08004000 - 0x0800BFFF  App Slot A       32 KiB  page 8-23
0x0800C000 - 0x08013FFF  App Slot B       32 KiB  page 24-39
0x08014000 - 0x0801CFFF  App data/scratch 36 KiB  page 40-57
0x0801D000 - 0x0801DFFF  Factory TLV       4 KiB  page 58-59
0x0801E000 - 0x0801EFFF  EEPROM            4 KiB  page 60-61
0x0801F000 - 0x0801FFFF  FEE               4 KiB  page 62-63
```

风险：32 KiB slot 很可能不够。若 slot 增大，data/scratch 会被压缩。

### 方案 B：片内单槽 + 外部暂存

适用条件：有外部 NOR Flash 或通信模块可可靠缓存固件。

```text
片内 Flash:
  Bootloader
  Active App
  Factory TLV
  EEPROM
  FEE

外部 Flash / 模块存储:
  OTA image staging
```

优点：片内 Flash 压力小。

缺点：依赖外部存储可靠性和驱动。

### 方案 C：无双槽，Bootloader 串流升级

适用条件：成本最低，允许升级失败后依赖 bootloader 重新下载。

流程：

```text
bootloader 接收固件 -> 边擦边写 app 区 -> 校验 -> 启动
```

优点：省 Flash。

缺点：升级中断时 app 可能不可用，必须保证 bootloader 有通信恢复能力。

### OTA 元数据

OTA 元数据不建议放 FEE。建议单独 page 或 bootloader 管理结构。

字段：

```text
active_slot
pending_slot
image_size
image_crc32
image_version
boot_attempts
rollback_flag
state: IDLE / DOWNLOADED / PENDING / CONFIRMED / ROLLBACK
```

如果只有一个 metadata page，需要有双记录或状态机，避免断电损坏。

推荐双 metadata record 或双 page：

```text
OTA Meta A
OTA Meta B
```

启动时选择 `CRC OK && seq 最大` 的元数据。

## 数据归属规则

| 数据 | 推荐位置 | 说明 |
|---|---|---|
| serial number | Factory TLV | 出厂身份，不应运行期覆盖原始值 |
| MMSI | Factory TLV | 如允许现场修改，用 FEE 保存 override |
| hardware revision | Factory TLV | 出厂固定 |
| RF/GNSS 校准 | Factory TLV | 出厂固定，整体 CRC |
| factory test flags | Factory TLV | 工厂写入 |
| default server | Factory TLV | 出厂默认值 |
| runtime server | FEE | 现场配置，覆盖 factory default |
| APN/MQTT topic | FEE | 低频配置 |
| report interval | FEE | 低频配置 |
| distress flags | FEE 或 EEPROM | 低频用 FEE，高频小状态用 EEPROM |
| boot_count | EEPROM | 高频小定长 |
| reset/error counters | EEPROM | 高频小定长 |
| event history | Evtlog | 固定记录日志 |
| OTA image | OTA slot/staging | 不放 FEE |
| OTA metadata | OTA metadata page | bootloader 管理 |

## 启动加载顺序

推荐启动时按以下顺序加载：

```text
1. Factory TLV
   读取 serial、MMSI、硬件版本、校准、默认服务器。

2. FEE
   读取 runtime server、APN、上报间隔、运行期 flags。

3. EEPROM
   更新 boot_count，读取小计数器和状态字。

4. Evtlog
   恢复事件日志写指针。
```

有效配置合成规则：

```text
effective_server = FEE.server if valid else FactoryTLV.default_server
effective_mmsi   = FEE.mmsi_override if valid else FactoryTLV.mmsi
```

不要修改 Factory TLV 来表达运行期覆盖。

## 当前落地状态

已完成：

- FEE Nano 硬件 Flash backend。
- FEE page header 状态机和 GC 事务边界。
- EEPROM 固定长度模拟组件。
- EEPROM page 状态恢复。
- PLB-N32 末尾 2 页 FEE。
- PLB-N32 倒数第 3-4 页 EEPROM。
- `boot_count` 使用 EEPROM。
- `server endpoint` 使用 FEE。
- PLB-N32 启动日志打印 FEE / EEPROM 初始化状态。

待完成：

- Factory TLV 分区定义。
- Factory TLV header + CRC32。
- Factory TLV 只读 API。
- Factory TLV 工厂写入工具或接口。
- Linker script 预留 factory / EEPROM / FEE 区。
- OTA bootloader 分区重规划。
- OTA metadata 状态机。
- 如启用 OTA 双槽，重新评估 App 最大体积。

## 实施优先级

1. 固化当前 EEPROM / FEE 分区，并在 linker script 中保留末尾 8 KiB。
2. 增加 Factory TLV 1 页或 2 页，只读解析 API。
3. 将 serial / MMSI / default server 从硬编码迁移到 Factory TLV。
4. 明确 OTA 方案，是片内双槽、外部暂存，还是 bootloader 串流升级。
5. 增加 OTA metadata 分区，不复用 FEE。
6. Bootloader 接管镜像校验、切换、回滚。

## 原则

- 工厂原始数据不放 FEE。
- 运行期配置不改 Factory TLV。
- 高频小状态不放 FEE。
- 固件镜像和 OTA metadata 不放 FEE。
- 裸 Flash 操作不直接暴露给业务保存参数。
- 所有跨重启数据必须有 CRC、版本或事务边界。
