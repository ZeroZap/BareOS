### 场景一：纯软件实现（无硬件加速）

这是最考验单片机性能的情况，通常用在没有加密硬件加速器的低端MCU上。

- **典型MCU**：STM32F103（Cortex-M3），ATmega2560（8位AVR），ESP8266（当不使用其AES部分进行优化时）。
- **占用情况**：
  - **Flash/ROM占用**：**大约 5KB - 10KB**。
    这包含了SHA256算法本身（约占3-6KB）和HMAC的构建逻辑。对于代码空间紧张的8位单片机（可能只有32KB Flash），这是一个不小的开销。
  - **RAM占用**：
    - **栈空间**：计算过程中的变量、中间状态等，需要几百字节。
    - **堆/静态内存**：主要是一个**64字节的块缓冲区**和一个**32字节的上下文结构体**。HMAC需要存储 `ipad` 和 `opad` 的密钥，这又需要2个64字节的块（共128字节）。
    - **总计**：**峰值RAM占用通常在 200 - 300 字节左右**。这对于只有2KB RAM的MCU来说压力很大，但对于拥有20KB RAM的Cortex-M0/M3来说完全可以接受。
  - **CPU时间/性能**：
    - **非常耗时**。SHA256需要对数据进行分组，每组进行64次循环运算，包含大量的逻辑运算和移位。
    - **粗略估算**：在 72MHz 的 Cortex-M3 上，计算1KB数据的HMAC-SHA256可能需要**几毫秒到十几毫秒**。在16MHz的8位AVR上，可能需要**几十甚至上百毫秒**。如果需要在低功耗模式下频繁计算，这会严重影响电池寿命。

**小结**：纯软件实现**代码体积大、计算速度慢**，但通用性强。适用于对性能要求不高、Flash和RAM资源相对宽裕的场合。

### Cortex-M0 纯软件 HMAC-SHA256 资源占用详情

#### 1. Flash (ROM) 占用

这是最主要的开销之一。

- **估算范围**：**约 2.5 KB - 5 KB**。
- **构成分析**：
  - **SHA256 核心算法**：这是大头。包含了消息调度、64轮循环运算、常量表等。由于M0指令集简单，每个操作都需要多条指令来完成，因此代码会被稍微“膨胀”，大约占 **1.5KB - 3KB**。
  - **HMAC 封装逻辑**：这部分代码很简单，就是处理密钥（如果过长则先哈希，过短则填充）、与 `ipad`/`opad` 异或、两次调用 SHA256 等。约占 **0.5KB - 1KB**。
  - **编译器影响**：使用 `-Os` (优化大小) 选项编译，可以显著减小代码体积，使其接近范围的下限。如果使用 `-O2` (优化速度)，代码体积可能会增大 10-20%。

**影响**：对于一款只有 32KB Flash 的典型 M0 MCU（如 STM32F0），这部分占用约占 **8%-15%**，是可接受的。但对于只有 16KB Flash 的型号，占用可能达到 **15%-30%**，就需要仔细权衡了。

#### 2. RAM 占用

RAM 占用通常比 Flash 更关键，因为低端 M0 的 RAM 往往非常小（4KB-8KB 很常见）。

- **估算范围**：**峰值占用约 200 - 350 字节**。
- **构成分析（核心）**：
  - **SHA256 上下文结构体**：通常包含：
    - `uint32_t state[8]`：存放哈希中间状态和最终结果。 --> **32 字节**
    - `uint8_t buffer[64]`：存放还未凑满一个块的数据。 --> **64 字节**
    - `uint64_t bit_count`：总位数计数。 --> **8 字节**
    - 其他标志位或索引 --> **约 4-8 字节**
    - **小计：~108 字节**
  - **HMAC 密钥处理**：
    - HMAC 需要将密钥与 `ipad` 和 `opad` 进行异或。通常需要一个 **64 字节** 的数组来存放处理后的密钥 `K0`。
    - **注意**：在计算过程中，可能需要两份这样的缓冲区（虽然可以复用，但栈帧峰值可能会包含它们）。
  - **栈空间**：函数调用、局部变量、参数传递等需要额外的栈空间。在执行 HMAC 计算时，调用栈深度增加，可能会额外消耗 **50-100 字节** 的栈。

**关键影响**：如果你的 MCU 只有 **4KB (4096 bytes) RAM**，那么 HMAC-SHA256 在运行时就会吃掉将近 **10%** 的RAM。这意味着在你的应用中，必须非常小心地管理剩余的堆和栈空间，避免堆栈溢出。

#### 3. CPU 性能和时序

这是纯软件实现最大的瓶颈。

- **性能估算**：
  - **平台**：以 48MHz 的典型 Cortex-M0（如 STM32G030）为例。
  - **计算一次 HMAC-SHA256 的时间**：
    - 计算时间与输入数据的长度直接相关。公式大致为：`时间 ≈ (数据长度 + 填充 + 2个块的开销) * 每块计算时间`。
    - **空数据（只有一个填充块）**：可能需要 **1-2 毫秒**。
    - **1KB (1024字节) 数据**：需要处理 `(1024 + 9) / 64 ≈ 17` 个数据块。耗时可能在 **15-25 毫秒** 量级。
- **实际影响**：
  - **功耗**：在这几十毫秒的时间里，CPU 需要全速运行，无法进入低功耗模式，对于电池供电设备来说是巨大的能耗。
  - **实时性**：如果你的应用有一个主循环需要快速响应外部事件（如按键、通信），那么一个长达 20ms 的 HMAC 计算会**严重阻塞**整个系统，感觉上就是“卡住了”。你可能需要采用分块计算的非阻塞方式，或者在计算期间临时关闭中断，这又会带来实时性问题。

### 总结与给开发者的建议

对于 **Cortex-M0** 上的纯软件 HMAC-SHA256，可以总结为：

- **Flash**：**可接受**。只要你的项目不是极度 Flash 紧缺，这 3-5KB 的代价是值得的。
- **RAM**：**需要警惕**。尤其是在 RAM < 8KB 的系统中，你必须确保在运行 HMAC 时，堆栈有足够的余量。
- **CPU/性能**：**主要瓶颈**。在需要处理大量数据或频繁计算时，性能问题会非常突出。

**应对策略：**

1. **选择优化库**：使用专为嵌入式系统设计的轻量级密码库，如 **TinyCrypt** 或 **μ加密库**。它们通常比通用的 mbedTLS 更节省资源。
2. **分块输入**：利用 SHA256 的流式特性，使用 `Update` 函数分多次输入数据，而不是一次性传入所有数据。这可以避免在 RAM 中存放整个大数据块。
3. **非阻塞设计**：如果性能瓶颈严重影响体验，考虑寻找带硬件加速的 M0+ 或 M3 内核的 MCU。如果必须用 M0，可以将 HMAC 计算放在一个低优先级的后台任务中，或者使用状态机将其拆分成多个步骤执行，避免长时间阻塞主循环。
4. **密钥预处理**：如果 HMAC 的密钥是固定的，可以预先计算好 `K0 ^ ipad` 和 `K0 ^ opad` 并存储在 Flash 中，这样可以节省每次计算时的 RAM 占用和一部分 CPU 时间。

**结论**：在 Cortex-M0 上实现纯软件的 HMAC-SHA256 **是可行的**，但开发者必须清醒地认识到其 **CPU 性能代价和 RAM 占用**，并在系统设计阶段就做好规划。
