### 核心区别：对称 vs. 非对称

- **对称加密（PSK属于此类）：**
  - **好比是一把真实的门锁和一把钥匙。** 你和你的室友都有同一把钥匙（共享的秘密）。
  - **加密和解密使用同一把钥匙。**

- **非对称加密（RSA/ECC属于此类）：**
  - **好比一个挂锁和它的钥匙。** 你可以买无数把这样的挂锁（这相当于你的**公钥**），但只有一把开锁的钥匙（这相当于你的**私钥**）。
  - **加密和解密使用不同的钥匙。**
  - **工作方式：** 你把打开的挂锁（公钥）发给全世界任何人。任何人想给你寄信，都可以把信放进盒子，用你的挂锁“咔哒”一声锁上。这个盒子一旦锁上，就只有你手里的那把唯一钥匙（私钥）才能打开。
  - **优势：** 你分发“挂锁”（公钥）的过程完全不需要保密，即使被截获也没关系，因为截获者无法打开已经锁上的盒子。

PSK就是在通信**开始之前**，通信双方通过一个**安全的外部渠道**预先共享的一个秘密。这个秘密通常是一个字符串（密码）。

- **例如：**
  - 你在路由器上设置Wi-Fi密码，然后在手机连接时输入它。这个密码就是PSK。
  - 你在配置两个服务器之间的VPN时，手动在两边配置一个相同的“预共享密钥”。

**2. 它如何工作？（以TLS-PSK为例）**

  3. 随后，双方**基于这个共享的PSK**，通过一系列算法，衍生出用于本次会话的对称加密密钥，开始加密通信。

**核心意思：** 一旦某人执行了一个动作（例如签署了一份文件或发起一笔交易），他/她**事后无法成功抵赖**，无法声称“这不是我干的”。

X25519 (ECDH) 如何进行密钥交换的，需要彼此有个公钥之类的嚒





我们使用X25519进行密钥交换时，确实需要彼此的公钥。X25519是椭圆曲线Diffie-Hellman（ECDH）密钥交换的一种，它基于 Curve25519 曲线。

下面是X25519密钥交换的步骤：

1. 生成密钥对：每个参与方生成自己的密钥对，包括一个私钥和一个公钥。
   - 私钥是一个随机数（在Curve25519中，是一个256位的随机数，但有一些位会被设置和清除以满足曲线要求）。
   - 公钥是通过将私钥与曲线上的基点相乘得到的。
2. 交换公钥：双方将自己的公钥发送给对方。注意，公钥可以公开传递，即使被第三方截获，没有私钥也无法计算出共享秘密。
3. 计算共享秘密：每个参与方使用自己的私钥和对方的公钥进行计算。具体来说，将自己的私钥与对方的公钥相乘（在曲线上标量乘法），得到的结果就是一个共享的秘密点。这个点的x坐标就是共享秘密（通常经过编码后是一个32字节的序列）。
4. 密钥派生：共享秘密可以直接用作对称密钥，或者通过一个密钥派生函数（KDF）来生成多个密钥（例如加密密钥和认证密钥）。

为什么这样安全？
因为即使攻击者知道了双方的公钥，也无法计算出共享秘密。这是因为在椭圆曲线上，已知公钥和基点，计算私钥是椭圆曲线离散对数问题（ECDLP），在Curve25519上被认为是计算上不可行的。

注意：在实际应用中，我们通常不会直接使用原始共享秘密作为密钥，而是将其输入到KDF中以得到更安全的密钥。同时，为了确保前向安全性，每次会话应该使用新的临时密钥对（ ephemeral key pairs ）。

### 核心前提：非对称密码学的力量

和所有非对称算法一样，X25519也基于**密钥对**的概念：

- **私钥**：一个保密的随机数（对于X25519，是一个256位的随机数）。
- **公钥**：由私钥通过一个**单向函数**计算得出。可以从私钥推导出公钥，但**不可能**从公钥反推出私钥。

------

### X25519密钥交换的详细步骤

假设Alice和Bob想要建立一个共享密钥。

#### **第1步：各自生成密钥对**

- **Alice**：
  - 随机生成她的私钥：`priv_a`
  - 计算她的公钥：`pub_a = X25519(priv_a, G)` （其中G是曲线上的一个固定基点）
- **Bob**：
  - 随机生成他的私钥：`priv_b`
  - 计算他的公钥：`pub_b = X25519(priv_b, G)`

#### **第2步：交换公钥**

- Alice将她的公钥 `pub_a` 发送给Bob。
- Bob将他的公钥 `pub_b` 发送给Alice。
- **关键点**：这些公钥可以在不安全的信道中传输。即使攻击者Eve截获了 `pub_a` 和 `pub_b`，她也无法从中推导出私钥 `priv_a` 或 `priv_b`。

#### **第3步：计算共享密钥（魔法发生的地方）**

这是最精妙的一步，双方**独立计算**，却能得到**完全相同的结果**。

- **Alice的计算**：
  - 她用自己的私钥 `priv_a` 和 Bob的公钥 `pub_b` 进行计算：
  - `shared_secret_alice = X25519(priv_a, pub_b)`
- **Bob的计算**：
  - 他用自己的私钥 `priv_b` 和 Alice的公钥 `pub_a` 进行计算：
  - `shared_secret_bob = X25519(priv_b, pub_a)`

#### **第4步：数学魔法为何有效？**

这之所以能工作，是因为椭圆曲线密码学的数学性质：**标量乘法满足交换律**。

让我们用简单的数学来表示（虽然实际是椭圆曲线运算，但原理类似）：

- `pub_a = priv_a * G`
- `pub_b = priv_b * G`

现在看双方的计算：

- Alice计算：`priv_a * pub_b = priv_a * (priv_b * G)`
- Bob计算：`priv_b * pub_a = priv_b * (priv_a * G)`

由于标量乘法满足交换律：
`priv_a * (priv_b * G) = priv_b * (priv_a * G) = (priv_a * priv_b) * G`

**因此，双方计算出了同一个值：`(priv_a \* priv_b) \* G`！**

这个值就是他们的`shared_secret`。

### 可视化类比

想象一种特殊的“混合颜料”：

1. **准备阶段**：
   - Alice有自己的**秘密颜料A**。
   - Bob有自己的**秘密颜料B**。
     -有一种**公共基础颜料G**。
2. **制作并交换公开混合物**：
   - Alice将基础颜料G和她的秘密颜料A混合，得到**公开混合物A**，发给Bob。
   - Bob将基础颜料G和他的秘密颜料B混合，得到**公开混合物B**，发给Alice。
3. **生成最终颜色**：
   - Alice将Bob的公开混合物B与自己的秘密颜料A混合。
   - Bob将Alice的公开混合物A与自己的秘密颜料B混合。
   - **奇迹**：双方最终得到了**完全相同的颜色**！

即使窃听者Eve截获了公开混合物A和B，她也无法分离出原始的秘密颜料A或B，因此无法复制出最终的混合颜色。

### 实际应用中的重要细节

#### **临时密钥对**

为了获得**前向保密**，在实际通信中（如TLS），双方通常会为**每次会话**生成一个**临时的**X25519密钥对。会话结束后就丢弃私钥。这样即使服务器长期私钥未来被泄露，过去的通信记录也无法被解密。

#### **密钥派生**

原始的`shared_secret`不能直接用作加密密钥。需要经过一个**密钥派生函数**（如HKDF）来生成实际使用的对称密钥。

#### **完整的握手流程**

一个典型的流程如下：

text

```
Alice                          Bob
 |                              |
 | 生成临时密钥对 (priv_a, pub_a) | 
 |--- pub_a ------------------->|
 |                              | 生成临时密钥对 (priv_b, pub_b)
 |                              | 计算 shared_secret = X25519(priv_b, pub_a)
 |<-- pub_b --------------------|
 | 计算 shared_secret = X25519(priv_a, pub_b) |
 |                              |
 | 双方现在拥有相同的 shared_secret          |
 | 使用KDF派生出加密密钥和MAC密钥            |
 |--- 开始加密和认证的通信 --------------->|
```



### 总结

**是的，X25519密钥交换确实需要交换公钥。**

但其安全性的核心在于：

1. **公钥可以公开传输**，不会危及安全。
2. **只有持有对应私钥的一方**才能利用收到的公钥计算出共享秘密。
3. 基于椭圆曲线离散对数问题的困难性，**攻击者无法从公钥推导出私钥**，也无法从双方的公钥计算出共享秘密。

这就是为什么X25519成为现代安全协议（如TLS 1.3、SSH、Signal等）中密钥交换的首选方案——它快速、安全且提供了完美的前向保密。

嵌入式如何单纯实现 密钥派生函数（如HKDF）





我们将在嵌入式系统中实现HKDF（HMAC-based Key Derivation Function）。由于嵌入式系统资源有限，我们可能需要自己实现HMAC，或者依赖已有的轻量级密码库。

步骤：

1. 选择哈希函数：通常使用SHA-256，但根据嵌入式系统的资源，也可以选择更轻量的哈希函数，但安全性会降低。这里我们以SHA-256为例。
2. 实现HMAC：如果系统没有提供HMAC，我们需要实现HMAC-SHA256。
3. 实现HKDF：按照RFC5869标准实现HKDF。

注意：嵌入式系统可能没有动态内存分配，因此我们尽量使用静态内存。

我们将分步骤实现：

第一步：实现HMAC-SHA256
第二步：实现HKDF

由于嵌入式系统的限制，我们假设不能使用大型标准库，因此我们将自己实现这些函数。

我们假设已经有一个可用的SHA-256实现。如果没有，我们需要先实现SHA-256。这里我们假设有一个名为`sha256`的函数，可以计算SHA-256哈希。

为了节省空间，我们不会实现完整的SHA-256，而是假设有一个现成的实现。如果你没有，可以参考一些轻量级的SHA-256实现。

下面我们开始：

首先，定义HMAC-SHA256函数：

c

```
#include <string.h>

// 假设你的SHA-256函数原型为：
void sha256(const unsigned char *data, size_t len, unsigned char *hash);

// 定义SHA-256的块大小（字节）和输出长度（字节）
#define SHA256_BLOCK_SIZE 64
#define SHA256_HASH_SIZE  32

// HMAC-SHA256计算函数
void hmac_sha256(const unsigned char *key, size_t key_len,
                 const unsigned char *data, size_t data_len,
                 unsigned char *output) {
    unsigned char k[SHA256_BLOCK_SIZE];
    unsigned char inner_hash[SHA256_HASH_SIZE];
    unsigned char outer_key[SHA256_BLOCK_SIZE];
    unsigned char inner_key[SHA256_BLOCK_SIZE];
    int i;

    // 如果key长度超过块大小，先对key进行hash，然后用hash值作为key
    if (key_len > SHA256_BLOCK_SIZE) {
        sha256(key, key_len, k);
        key_len = SHA256_HASH_SIZE;
    } else {
        memcpy(k, key, key_len);
    }

    // 如果key长度不足块大小，则用0填充
    if (key_len < SHA256_BLOCK_SIZE) {
        memset(k + key_len, 0, SHA256_BLOCK_SIZE - key_len);
    }

    // 生成inner key和outer key
    for (i = 0; i < SHA256_BLOCK_SIZE; i++) {
        inner_key[i] = k[i] ^ 0x36;
        outer_key[i] = k[i] ^ 0x5C;
    }

    // 计算inner hash: inner_key || data
    unsigned char inner_data[SHA256_BLOCK_SIZE + data_len];
    memcpy(inner_data, inner_key, SHA256_BLOCK_SIZE);
    memcpy(inner_data + SHA256_BLOCK_SIZE, data, data_len);
    sha256(inner_data, SHA256_BLOCK_SIZE + data_len, inner_hash);

    // 计算outer hash: outer_key || inner_hash
    unsigned char outer_data[SHA256_BLOCK_SIZE + SHA256_HASH_SIZE];
    memcpy(outer_data, outer_key, SHA256_BLOCK_SIZE);
    memcpy(outer_data + SHA256_BLOCK_SIZE, inner_hash, SHA256_HASH_SIZE);
    sha256(outer_data, SHA256_BLOCK_SIZE + SHA256_HASH_SIZE, output);
}
```



第二步：实现HKDF

HKDF有两个步骤：提取（Extract）和扩展（Expand）。我们按照RFC5869实现。

函数原型：

```
// HKDF提取阶段：使用盐和输入密钥材料（IKM）生成伪随机密钥（PRK）
void hkdf_extract(const unsigned char *salt, size_t salt_len,
                  const unsigned char *ikm, size_t ikm_len,
                  unsigned char *prk) {
    // 如果盐为空，则使用一个全0的盐，长度等于哈希输出长度
    unsigned char zero_salt[SHA256_HASH_SIZE];
    if (salt == NULL || salt_len == 0) {
        memset(zero_salt, 0, SHA256_HASH_SIZE);
        salt = zero_salt;
        salt_len = SHA256_HASH_SIZE;
    }
    hmac_sha256(salt, salt_len, ikm, ikm_len, prk);
}

// HKDF扩展阶段：使用PRK、信息和输出长度生成输出密钥材料（OKM）
void hkdf_expand(const unsigned char *prk, size_t prk_len,
                 const unsigned char *info, size_t info_len,
                 unsigned char *okm, size_t okm_len) {
    unsigned char T[SHA256_HASH_SIZE];
    unsigned char Tn[SHA256_HASH_SIZE];
    unsigned char loop_data[info_len + 1];
    size_t n = (okm_len + SHA256_HASH_SIZE - 1) / SHA256_HASH_SIZE;
    size_t i, current_len;

    if (n > 255) {
        // 错误处理：输出长度太长
        return;
    }

    for (i = 1; i <= n; i++) {
        // 计算T(1), T(2), ... T(n)
        if (i == 1) {
            // T1 = HMAC(PRK, T(0) || info || i)  其中T(0)为空
            memcpy(loop_data, info, info_len);
            loop_data[info_len] = i;
            hmac_sha256(prk, prk_len, loop_data, info_len + 1, T);
        } else {
            // T(i) = HMAC(PRK, T(i-1) || info || i)
            memcpy(loop_data, Tn, SHA256_HASH_SIZE);
            memcpy(loop_data + SHA256_HASH_SIZE, info, info_len);
            loop_data[SHA256_HASH_SIZE + info_len] = i;
            hmac_sha256(prk, prk_len, loop_data, SHA256_HASH_SIZE + info_len + 1, T);
        }
        memcpy(Tn, T, SHA256_HASH_SIZE);

        // 将T(i)复制到OKM中
        current_len = (i == n) ? (okm_len - (i-1)*SHA256_HASH_SIZE) : SHA256_HASH_SIZE;
        memcpy(okm + (i-1)*SHA256_HASH_SIZE, T, current_len);
    }
}

// 完整的HKDF函数
void hkdf(const unsigned char *salt, size_t salt_len,
          const unsigned char *ikm, size_t ikm_len,
          const unsigned char *info, size_t info_len,
          unsigned char *okm, size_t okm_len) {
    unsigned char prk[SHA256_HASH_SIZE];
    hkdf_extract(salt, salt_len, ikm, ikm_len, prk);
    hkdf_expand(prk, SHA256_HASH_SIZE, info, info_len, okm, okm_len);
}
```

```
    uint32_t count[2];
    uint8_t buffer[64];

void sha256_update(sha256_ctx_t *ctx, const uint8_t *data, size_t len);
void sha256_final(sha256_ctx_t *ctx, uint8_t digest[32]);

// HMAC-SHA256计算
void hmac_sha256(const uint8_t *key, size_t key_len,
                const uint8_t *data, size_t data_len,
                uint8_t out[32]) {
    sha256_ctx_t ctx;
    uint8_t k_ipad[64], k_opad[64];
    uint8_t tmp_key[32];
    
    // 如果密钥超过块大小，先哈希压缩
    if (key_len > 64) {
        sha256_init(&ctx);
        sha256_update(&ctx, key, key_len);
        sha256_final(&ctx, tmp_key);
        key = tmp_key;
        key_len = 32;
    }
    
    // 准备内填充和外填充密钥
    memset(k_ipad, 0x36, 64);
    memset(k_opad, 0x5C, 64);
    
    for (size_t i = 0; i < key_len; i++) {
        k_ipad[i] ^= key[i];
        k_opad[i] ^= key[i];
    }
    
    // 内哈希
    sha256_init(&ctx);
    sha256_update(&ctx, k_ipad, 64);
    sha256_update(&ctx, data, data_len);
    sha256_final(&ctx, out);
    
    // 外哈希
    sha256_init(&ctx);
    sha256_update(&ctx, k_opad, 64);
    sha256_update(&ctx, out, 32);
    sha256_final(&ctx, out);
}
```

```
               const uint8_t *info, size_t info_len,
```

## 核心澄清：什么是"共享秘密"

**共享秘密** 是在密钥交换过程中**计算出来的，而不是传输的**。

在X25519密钥交换中：

- **交换的是公钥**（公开的，可以传输）
- **计算出来的是共享秘密**（保密的，从不传输）

## 为什么这个过程安全？

1. **前向保密**：每次会话使用新的临时密钥对
2. **中间人无法计算共享秘密**：
   - 攻击者可以看到 `client_public` 和 `server_public`
   - 但没有 `client_private` 或 `server_private`
   - 因此无法计算出 `shared_secret`
3. **数学保证**：椭圆曲线离散对数问题确保从公钥推导私钥在计算上不可行

## 为什么考虑使用认证加密（AEAD）？

1. **效率更高**：AEAD模式在一个算法中同时完成加密和认证，这通常比先加密再计算MAC（如HMAC）的方式更快，因为只需要一遍处理数据。
2. **更少的代码和复杂性**：使用一个AEAD算法而不是两个独立的算法（加密和MAC）可以减少代码量，降低实现复杂性，从而减少出错的可能性。
3. **安全性证明**：许多AEAD模式（如AES-GCM）有严格的安全证明，而组合使用加密和MAC需要谨慎，否则可能导致安全漏洞。例如，如果加密和MAC使用相同的密钥，或者MAC没有覆盖密文，可能会出现问题。
4. **标准化和广泛采用**：AEAD模式已经成为TLS 1.2和1.3等现代协议的标准，得到了广泛的审查和采用。
5. **侧信道攻击抵抗**：一些AEAD模式（如AES-GCM）在实现时可能更容易抵抗侧信道攻击，但这一点取决于具体实现。

## AES-GCM和AES-CCM简介

### AES-GCM (Galois/Counter Mode)

AES-GCM是一种广泛使用的AEAD模式，它结合了CTR模式的加密和Galois模式的身份验证。

- **加密部分**：使用AES-CTR模式生成密钥流，与明文异或得到密文。
- **认证部分**：使用Galois域上的乘法来计算认证标签（MAC），该标签同时保护密文和附加数据（Additional Authenticated Data, AAD）。

**优点**：

- 并行计算，高速率。
- 硬件加速支持（如Intel AES-NI和CLMUL指令集）。

**缺点**：

- 对IV（初始化向量）的重复使用非常敏感，如果IV重复，会导致密钥流重复，从而可能泄露明文和认证密钥。
- 在软件实现中，认证部分可能容易受到侧信道攻击。

### AES-CCM (Counter with CBC-MAC)

AES-CCM是另一种AEAD模式，它结合了CTR模式的加密和CBC-MAC认证。

- **认证部分**：首先使用CBC-MAC计算明文的MAC。
- **加密部分**：使用CTR模式加密明文和MAC。

**优点**：

- 基于CBC-MAC，结构简单。
- 对IV重复使用的敏感性较低（但仍然需要避免重复）。

**缺点**：

- 串行操作，无法并行化，因此速度较慢。
- 需要两次处理数据（一次用于CBC-MAC，一次用于CTR加密），效率较低。

## 对比加密+HMAC与AEAD

| 特性         | 加密+HMAC                                       | AEAD (如AES-GCM)     |
| :----------- | :---------------------------------------------- | :------------------- |
| 速度         | 较慢（两次处理数据）                            | 较快（一次处理数据） |
| 代码大小     | 较大（两个算法）                                | 较小（一个算法）     |
| 安全性       | 需要谨慎组合（如使用不同密钥，确保MAC覆盖密文） | 内置安全组合         |
| IV/Nonce管理 | 需要两个不同的IV（加密和MAC）                   | 只需要一个Nonce      |
| 标准化       | 是，但组合方式有多种                            | 是，单一标准         |

## 在IoT中的考虑

在资源受限的IoT设备中，AEAD通常更受欢迎，因为：

- 减少计算量，节省电量和时间。
- 减少代码体积，节省存储空间。

但是，需要注意的是，AES-GCM对Nonce的重用非常敏感，因此在IoT设备中必须确保Nonce的唯一性（通常使用计数器或随机数）。如果无法保证Nonce的唯一性，可以考虑使用AES-CCM，但AES-CCM的性能较低。

## 总结

在IoT中，从加密+HMAC转向AEAD（如AES-GCM或AES-CCM）可以带来性能、代码大小和安全性方面的好处。但是，需要根据具体设备的能力和安全需求来选择。如果设备支持硬件加速，AES-GCM可能是更好的选择；如果设备资源非常有限且无法保证Nonce的唯一性，AES-CCM可能更合适。

```
    // ... HMAC初始化步骤

void hmac_stream_update(hmac_stream_ctx_t *ctx, 
                       const uint8_t *data, size_t len) {
    // 分块处理大数据，避免大缓冲区
    while (len > 0) {
        size_t chunk_len = (len < 512) ? len : 512;
        sha256_update(&ctx->sha_ctx, data, chunk_len);
        data += chunk_len;
        len -= chunk_len;
    }
}

void hmac_stream_final(hmac_stream_ctx_t *ctx, uint8_t hmac_tag[32]) {
    uint8_t hash[32];
    
    // 完成SHA256
    sha256_final(&ctx->sha_ctx, hash);
    
    // 计算HMAC（使用标准HMAC步骤）
    hmac_final(ctx->hmac_key, hash, hmac_tag);
}
```
