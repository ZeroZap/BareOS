我来为你设计一个兼容RTOS和裸机的AT Client框架。这个框架支持多设备、ringbuffer数据源和系统tick超时处理。

## 主要特性

1. **平台兼容性**
   - 通过宏定义区分RTOS和裸机环境
   - 互斥锁抽象，支持RTOS信号量和裸机空操作
   - 延时函数抽象
2. **多设备支持**
   - 可注册多个AT设备
   - 每个设备独立的ringbuffer和回调函数
   - 设备间独立状态管理
3. **数据流处理**
   - 基于回调的字节读取接口
   - Ringbuffer数据源支持
   - 行缓冲解析
4. **超时处理**
   - 使用系统tick计算超时
   - 支持tick溢出处理
   - 可配置超时时间
5. **响应处理**
   - 灵活的回调处理器
   - 支持OK/ERROR/自定义响应
   - 响应缓冲和溢出检测

## 使用建议

1. 根据实际需求调整配置参数（缓冲区大小、最大设备数等）
2. 在裸机环境下实现`AT_DELAY_MS`宏
3. 确保`get_tick`函数的时间单位一致（建议毫秒）
4. 在UART接收中断中填充ringbuffer
5. 复杂AT协议可扩展响应处理器

这个框架提供了良好的扩展性，可以根据具体AT模块的协议要求扩展响应处理器，支持复杂的交互流程。


支持动态urc注入解析

我们将在AT客户端框架中增加对动态URC（Unsolicited Result Code）注入解析的支持。URC是模块主动上报的信息，如来电、短信、网络状态变化等。我们需要能够动态注册URC处理函数，并在读取数据时进行匹配和处理。

主要修改点：

1. 添加URC处理函数的结构体，包含URC前缀字符串和处理回调函数。
2. 在每个AT设备中维护一个URC处理列表（动态数组或链表）。
3. 在读取数据时，除了等待命令响应外，还要检查是否有URC上报，并调用对应的处理函数。
4. 提供URC注册和注销的接口。

注意：由于URC是主动上报，可能发生在任何时候，因此我们需要在读取数据时（比如在at_readline函数中）检查是否匹配URC。但是，在等待命令响应时，我们仍然需要优先处理命令响应，同时也要处理URC（因为URC可能随时出现）。然而，在等待命令响应时，我们可能需要区分当前行是命令响应还是URC。通常，URC有特定的前缀，我们可以通过匹配前缀来区分。

设计思路：
我们将URC处理函数注册到AT设备中，当读取到一行数据时，首先检查是否匹配任何已注册的URC前缀，如果匹配，则调用相应的URC处理函数，然后继续读取下一行。否则，将这一行作为命令响应的一部分，交给命令响应处理器处理。

但是，在等待命令响应时，我们可能不希望URC处理函数干扰到命令响应的处理，因为URC处理函数可能会修改一些全局状态，而命令响应处理器可能不希望这样。因此，我们需要考虑是否允许在命令等待期间处理URC。通常，URC应该在任何时候都被处理，所以我们允许在命令等待期间处理URC。

修改at_readline函数，使其在读取到一行数据时，先检查URC列表，如果匹配则调用URC处理函数，并返回一个特殊值（比如0，表示这一行已经被处理，不作为命令响应）。但是，这样会改变at_readline的行为，可能需要重新设计。

另一种做法：在at_process_response函数中，每读取一行，先检查URC，如果是URC则处理，然后继续读取，直到读取到命令响应（或超时）。这样，URC的处理不会影响命令响应，因为URC被立即处理，而命令响应处理器只会看到非URC的行。

我们选择第二种做法，因为这样不会改变at_readline的语义，而且URC处理是即时性的。

步骤：

1. 定义URC处理结构体。
2. 在AT设备结构体中添加URC列表。
3. 实现URC注册和注销函数。
4. 修改at_process_response函数，在读取每一行后检查是否为URC，如果是则调用URC处理函数，并继续读取下一行。

注意：URC处理函数可能会做很多事情，甚至可能长时间阻塞，因此建议URC处理函数尽量简单，避免影响AT命令的响应。同时，URC处理函数中不能调用AT命令发送函数，以免造成死锁（因为设备可能正在等待命令响应）。

另外，URC处理函数可能需要在多个设备间共享，因此我们允许将同一个URC处理函数注册到多个设备。

由于URC列表可能不会太长，我们使用动态数组来存储URC处理结构体。

考虑到内存分配，我们可以在AT设备初始化时固定URC的最大数量，或者动态分配。这里为了简单，我们使用动态数组，并设定一个最大值（比如每个设备最多10个URC处理函数）。

但是，为了更灵活，我们使用链表来存储URC处理结构体，这样数量不受限制。但考虑到嵌入式环境，我们使用静态数组（固定最大数量）来避免动态内存分配。

我们选择使用静态数组，并在AT设备结构体中包含一个URC处理数组和一个URC数量字段。

具体修改at_process_response函数：

在读取一行后，遍历URC表，检查当前行是否以某个URC前缀开头，如果是，则调用对应的处理函数，然后跳过这一行，继续读取下一行。

这样，我们就可以在等待命令响应的同时处理URC。

另外，我们还需要在没有任何命令在等待响应时，也能够处理URC。因此，我们需要一个后台任务来读取并处理URC。但是，在没有命令时，设备可能没有在读取数据，所以我们需要主动去读取。

因此，我们还需要一个函数，用于在没有命令时处理URC（即主动读取并处理URC）。我们可以提供一个函数at_process_urc，它不断读取数据并处理URC，直到没有数据可读。然后，我们可以在主循环中调用这个函数，或者创建一个任务来调用。

但是，我们的框架是兼容裸机和RTOS的，所以我们需要考虑两种环境。

在RTOS中，我们可以创建一个任务来不断调用at_process_urc。在裸机中，我们可以在主循环中调用。

我们设计一个函数：at_device_process_urc(at_device_t *dev)，它读取一行数据，如果是URC则处理，否则丢弃（因为此时没有命令在等待响应，所以非URC的数据可能是噪声或者之前的残留）。然后，我们不断读取，直到没有数据可读。

但是，这样可能会一直读取，导致其他任务无法执行。所以，我们限制每次调用的最大读取行数，比如10行。

我们提供两个函数：

1. at_device_process_urc(at_device_t *dev, uint32_t timeout);
   在指定的超时时间内，不断读取并处理URC，直到超时。这个函数可以用于在后台任务中调用。
2. at_process_all_urc(uint32_t timeout);
   处理所有设备的URC。

但是，由于我们的AT设备可能在不同的任务中被使用，我们需要考虑线程安全。因此，我们在处理URC时也需要加锁。

然而，URC处理可能是在没有命令发送时，所以我们加锁时不能长时间等待。我们可以使用非阻塞锁，或者设计为在URC处理函数中不加锁，而在注册和注销URC时加锁。

我们选择在at_device_process_urc函数中加锁，因为读取设备数据需要独占设备。但是，如果设备正在发送命令（已经加锁），那么at_device_process_urc函数就会阻塞，直到命令发送完成。这可能会导致URC处理延迟。所以，我们使用非阻塞锁，如果设备正忙，就跳过URC处理。

修改at_device_process_urc函数，使用非阻塞方式加锁。

但是，在RTOS中，非阻塞锁可能不是直接支持，我们可以使用xSemaphoreTake(dev->mutex, 0)来实现非阻塞。

我们修改at_device_process_urc函数如下：

先尝试加锁，如果加锁失败，则直接返回，不处理URC。

然后，我们读取一行数据，如果读取到，就处理URC，然后继续读取，直到超时或者没有数据。

注意：在裸机环境下，我们可能没有非阻塞锁，所以我们可以直接处理，因为裸机是单任务的。

因此，我们修改框架，在裸机环境下，at_device_process_urc函数不加锁，而是直接读取。

但是，为了统一，我们在裸机环境下，将AT_MUTEX_LOCK和AT_MUTEX_UNLOCK定义为空，并且我们可以定义AT_MUTEX_TRY_LOCK为true（表示一直可以获取锁）。或者，我们单独为裸机定义一个AT_MUTEX_TRY_LOCK宏。

由于时间关系，我们暂不考虑裸机的非阻塞锁，因为裸机没有多任务，不需要锁。我们只在RTOS环境下实现非阻塞锁。
