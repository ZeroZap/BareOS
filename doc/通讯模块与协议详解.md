版本：v1.0-draft

定位：独立的业界通讯模块、协议体系与 MCU 接口参考文档  
用途：用于 IoT、裸机 MCU、应急救援、海事、工业、车辆、户外设备等产品在通信方案选型、固件架构、测试认证和协议实现时参考。

## 1. 文档边界

本文不绑定任何单一产品版本、BOM 或项目定义。本文关注三个层面：

| 层面 | 说明 | 示例 |
|---|---|---|
| 通讯模块 | 可接入 MCU 的无线/有线/声学/光学通信模块 | Cat1、NB-IoT、LoRa、BLE、AIS、RS-485 |
| 通讯协议 | 模块使用的空口协议、链路协议、应用协议、数据格式 | LTE、LoRaWAN、MQTT、NMEA、Modbus、CANopen |
| MCU 接口 | MCU 与模块之间的电气接口、控制方式和驱动模型 | UART AT、SPI、I2C、USB、CAN、RMII、GPIO |

本文的核心问题：

| 问题 | 回答方向 |
|---|---|
| 业界有哪些常见通讯模块？ | 按蜂窝、卫星、LPWAN、短距无线、海事、航空、工业总线、声学、光学分类 |
| MCU 怎么接这些模块？ | UART AT、SPI、I2C、USB、SDIO、PCIe、CAN、RMII、GPIO、I2S 等 |
| 每种协议适合什么场景？ | 从覆盖、功耗、速率、成本、认证、生态接收端分析 |
| 固件如何抽象？ | 通用设备驱动、AT 事务、异步 URC、协议帧、事件调度 |
| 有哪些标准文档要查？ | 3GPP、IEEE、IETF、ETSI、ITU、IEC、NMEA、LoRa Alliance、Bluetooth SIG 等 |

### 2.2 选型维度

| 维度 | 需要判断的问题 |
|---|---|
| 覆盖 | 是否依赖运营商、网关、岸站、卫星、救援队接收机？ |
| 功耗 | 是否支持深睡、PSM、硬断电、外部唤醒？ |
| 速率 | 是短报文、传感器数据、语音、图片还是视频？ |
| 时延 | SOS、遥控和语音需要低时延；表计可容忍分钟级时延 |
| 双向能力 | 是否需要 ACK、配置下发、OTA、远程控制？ |
| 生态接收端 | 手机、船舶、救援队、基站、卫星、网关是否能接收？ |
| 法规认证 | 是否涉及频率许可、型号核准、运营商入网、海事/航空认证？ |
| 天线 | 低频长天线、卫星天线、VHF 天线、UWB 天线是否能放进结构？ |
| MCU 资源 | 裸机 MCU 是否能承载 USB Host、TLS、IP 栈、文件系统？ |
| 模块接口 | UART AT 最简单；USB/PCIe/SDIO/RMII 更复杂但速率更高 |
| 成本 | 模组、天线、认证、流量、卫星资费、平台服务都要计算 |
| 可测试性 | 是否有射频测试模式、RSSI/SNR 查询、环回、日志、产测命令？ |

## 3. MCU 与通讯模块接口

### 3.1 接口总览

| MCU 接口 | 常接模块 | 优点 | 风险/限制 |
|---|---|---|---|
| UART | 蜂窝、GNSS、卫星、BLE、LoRa 模组、调试口 | 简单、裸机友好、AT 生态成熟 | 速率有限，异步 URC 解析复杂 |
| SPI | LoRa/UWB/RF 芯片、Flash、部分 BLE/Wi-Fi 芯片 | 高速、时序可控、驱动简单 | 需要片选管理和中断配合 |
| I2C | 传感器、燃料计、NFC、安全芯片、部分低速模块 | 引脚少、多设备共享 | 速度低，总线挂死需恢复 |
| USB Device | 调试、日志、DFU、虚拟串口 | PC 生态好 | 需要 USB 协议栈 |
| USB Host | 4G/5G/Wi-Fi 模组、U 盘 | 高速、标准类驱动多 | 裸机实现复杂，功耗高 |
| SDIO | Wi-Fi、SD 卡 | 高吞吐 | MCU 外设和驱动复杂 |
| PCIe | 高速 5G 模组 | 高速率 | 小 MCU 通常不可用 |
| CAN | 车辆、船舶、工业设备 | 抗干扰、仲裁、实时性好 | 需要收发器和协议栈 |
| RMII/MII | Ethernet PHY | 标准 IP 网络 | MCU 需 MAC、RAM 和网络栈 |
| GPIO | PWRKEY、RESET、IRQ、STATUS、片选 | 必需辅助控制 | 时序依赖厂商资料 |
| ADC/DAC/PWM/I2S | 音频、声学、红外、蜂鸣器、模拟 RF | 可做低层信号处理 | MCU 负担重，模拟链路复杂 |

### 3.2 UART AT 模型

AT 指令是 MCU 控制蜂窝、卫星、BLE、Wi-Fi、GNSS、LoRa 模组的常见方式。

```text
MCU -> Module: AT+CMD=param\r
Module -> MCU: \r\nOK\r\n
Module -> MCU: \r\n+URC: event,param\r\n
Module -> MCU: \r\nERROR\r\n
Module -> MCU: prompt 或二进制透传
```

AT 驱动必须支持：

| 能力 | 说明 |
|---|---|
| 命令事务 | 每个命令有发送、期望响应、超时、重试、失败回调 |
| URC 分流 | 主动上报可插入任何时刻，不能污染命令响应 |
| 多行响应 | 网络查询、GPS 查询、短信、扫描列表通常是多行 |
| Prompt 模式 | SMS、HTTP body、MQTT publish 可能先返回 `>` 再发 payload |
| 透传模式 | TCP/UDP/PPP/数据流会进入非行文本模式 |
| 二进制安全 | 协议不能假设数据永远是 ASCII 行 |
| 断电恢复 | 模组复位后清空事务、丢弃旧 buffer、重新同步 |

AT 状态机示例：

```text
IDLE
  -> SEND_CMD
  -> WAIT_ECHO optional
  -> WAIT_PROMPT optional
  -> SEND_PAYLOAD optional
  -> WAIT_INTERMEDIATE optional
  -> WAIT_FINAL OK/ERROR/TIMEOUT
  -> DONE/FAIL

Any state:
  incoming line matches URC -> dispatch_urc -> continue previous state
```

### 3.3 SPI/RF 芯片模型

SPI 常用于直接控制射频芯片，而不是完整模组。

```text
MCU SPI -> RF registers/FIFO
MCU GPIO -> NSS/RESET/TXEN/RXEN
RF IRQ -> MCU EXTI
```

适合：LoRa 芯片、Sub-GHz FSK、UWB、2.4 GHz 收发器、AIS/VHF RF 芯片、外部 Flash。

驱动重点：

| 项 | 说明 |
|---|---|
| 寄存器表 | 初始化、频率、功率、调制、包格式、IRQ mask |
| FIFO | TX/RX buffer、长度、CRC、白化、前导码 |
| IRQ | TX done、RX done、timeout、CRC error、CAD done |
| 校准 | TCXO、PLL、频偏、RSSI、PA ramp |
| 低功耗 | sleep、standby、listen、wake-on-radio |

### 3.4 通用驱动抽象

```c
typedef struct {
    int (*power_on)(void *ctx);
    int (*power_off)(void *ctx);
    int (*reset)(void *ctx);
    int (*configure)(void *ctx, const void *cfg);
    int (*send)(void *ctx, const uint8_t *buf, uint16_t len);
    int (*recv)(void *ctx, uint8_t *buf, uint16_t max_len);
    int (*process)(void *ctx, uint32_t now_ms);
    int (*get_status)(void *ctx, void *status);
} comm_device_ops_t;
```

建议把“接口层”和“协议层”分开：

| 层 | 职责 |
|---|---|
| port | UART/SPI/I2C/CAN/USB/GPIO 的读写和中断 |
| device | 模组上电、复位、初始化、状态查询 |
| protocol | AT、NMEA、LoRaWAN、AIS、Modbus、MQTT 等解析 |
| service | SOS、心跳、OTA、配置、日志、产测等业务 |

## 4. 蜂窝公网通信

### 4.1 制式对比

| 制式 | 特点 | 典型接口 | 适合场景 | 注意事项 |
|---|---|---|---|---|
| 2G GSM/GPRS | 低速，历史覆盖广 | UART AT | 存量设备 | 多国退网，不建议新项目依赖 |
| 3G | 中低速 | UART/USB | 存量设备 | 多地退网 |
| LTE Cat1 | 成本和速率平衡 | UART AT/USB | IoT、定位器、报警器 | 功耗高于 NB-IoT/LTE-M |
| LTE Cat4 | 较高速 | USB/PCIe | 视频、网关、车载 | 成本和功耗更高 |
| NB-IoT | 深覆盖、低功耗、低速 | UART AT | 表计、低频上报 | 移动性、实时性、语音能力弱 |
| LTE-M/Cat-M1 | 低功耗、移动性较好 | UART AT | 可穿戴、资产追踪 | 区域生态差异大 |
| 5G NR | 高速低时延 | USB/PCIe | 视频、工业网关 | 小电池设备不友好 |
| 5G RedCap | 轻量化 5G | USB/PCIe/UART | 中速工业 IoT | 新兴生态 |
| 3GPP NTN | 蜂窝卫星扩展 | AT/厂商 API | 偏远地区短消息 | 运营商和终端成熟度决定可用性 |

### 4.2 上层协议

| 协议 | 用途 | 适合 MCU 的方式 |
|---|---|---|
| MQTT/MQTTS | 遥测、状态、命令、长连接 | 模组内 MQTT 或 MCU 端轻量 MQTT |
| MQTT-SN | 低功耗短消息、网关场景 | UDP/串口网关 |
| HTTP/HTTPS | REST API、配置、OTA manifest | 模组内 HTTP(S) 最省 MCU 资源 |
| CoAP/DTLS | 低功耗 REST 风格 IoT | MCU 端实现可控但需 DTLS |
| LwM2M | 设备管理、FOTA、参数 | 商用 IoT 平台常用 |
| TCP/UDP 私有协议 | 极致低开销 | MCU 端状态机简单但平台自建成本高 |
| SMS PDU | 弱数据兜底 | 需处理编码、长短信、状态报告 |

## 6. LPWAN 与低功耗远距离通信

### 6.1 技术对比

| 技术 | 频段 | 网络形态 | 优点 | 缺点 |
|---|---|---|---|---|
| LoRa P2P | Sub-GHz | 点对点/私有星型 | 远距离、低功耗、自建简单 | 无统一公网生态 |
| LoRaWAN | Sub-GHz | 终端-网关-服务器 | 标准化、低功耗、安全机制完整 | 依赖网关覆盖 |
| Sigfox | Sub-GHz | 运营商网络 | 超低速、低功耗 | 覆盖和生态受限 |
| Wi-SUN | Sub-GHz | Mesh/IPv6 | 适合城市基础设施 | 栈复杂 |
| MIOTY | Sub-GHz | 星型/网关 | 抗干扰、低功耗 | 生态较小 |
| Wireless M-Bus | Sub-GHz | 表计网络 | 表计成熟 | 通用救援意义弱 |
| Z-Wave | Sub-GHz | 智能家居 Mesh | 家居生态 | 非广域救援链路 |

### 6.2 LoRa P2P

LoRa P2P 适合自定义低速信标、救援队自建接收器、山区或矿区局部覆盖。

MCU 接口：

| 芯片形态 | MCU 接口 |
|---|---|
| SX126x/SX127x 类 RF 芯片 | SPI + DIO IRQ + RESET + BUSY |
| LoRa AT 模组 | UART AT + GPIO |

典型参数：

| 参数 | 说明 |
|---|---|
| SF | Spreading Factor，越大距离越远、速率越低 |
| BW | 带宽，常见 125/250/500 kHz |
| CR | 编码率，影响抗干扰和速率 |
| TX Power | 受法规限制 |
| Duty Cycle | 部分地区限制占空比 |

### 6.3 LoRaWAN

LoRaWAN 架构：

```text
End Device -> Gateway -> Network Server -> Application Server
```

核心概念：

| 概念 | 说明 |
|---|---|
| Class A | 最低功耗，上行后打开下行窗口 |
| Class B | 有同步信标，周期下行窗口 |
| Class C | 基本持续接收，功耗高 |
| OTAA | 入网激活，使用 Join Server |
| ABP | 预激活，配置固定会话参数 |
| MIC | 报文完整性校验 |
| Frame Counter | 防重放 |

## 7. 短距无线通信

### 7.1 BLE

BLE 适合手机配置、近场发现、低功耗广播、维护和产测。

| 层 | 关键内容 |
|---|---|
| PHY | 1M、2M、LE Coded S=2/S=8 |
| GAP | 广播、扫描、连接、角色 |
| GATT | 服务、特征、通知、读写 |
| SMP | 配对、加密、绑定 |
| Mesh | 多节点转发，适合照明/楼宇，不适合远距离稀疏野外 |

MCU 接口：UART HCI、SPI HCI、完整 BLE SoC 通过 UART/SPI 自定义协议、SoC 内部应用。

### 7.2 Zigbee 与 Thread

| 技术 | 基础 | 特点 | 适用 |
|---|---|---|---|
| Zigbee | IEEE 802.15.4 + Zigbee 网络层/ZCL | 成熟 Mesh，低功耗 | 家居、楼宇、短距传感器 |
| Thread | IEEE 802.15.4 + IPv6/6LoWPAN | IP 化 Mesh | Matter、智能家居、楼宇 |

Zigbee/Thread 不适合没有基础设施的广域救援主链路，但适合团队、营地、舱内、楼宇内短距组网。

### 7.3 Wi-Fi

| 模式 | 用途 |
|---|---|
| STA | 连接路由器上传数据、OTA、日志 |
| AP | 设备开热点供手机配置 |
| Wi-Fi Direct | 手机/设备点对点连接 |
| Wi-Fi Aware | 近场发现 |

MCU 接口：UART AT、SPI、SDIO、USB。裸机 MCU 优先选 UART AT Wi-Fi 模组；高吞吐才考虑 SDIO/USB。

### 7.4 NFC/RFID

NFC 适合 cm 级身份读取、配置、维修追溯、无电标签。不适合远程通信。

| 类型 | 说明 |
|---|---|
| ISO/IEC 14443 | 近距离卡和读写器 |
| ISO/IEC 15693 | 较远距离 13.56 MHz 标签 |
| NFC Forum Tag | 手机可读标签格式 |
| Card Emulation | 设备模拟卡 |

### 7.5 UWB

UWB 适合高精度测距和近场定位。

## 15. 通讯编码格式、安全与数据格式

### 15.1 安全协议

| 技术 | 用途 |
|---|---|
| TLS/mTLS | 蜂窝/Wi-Fi 平台通信安全 |
| DTLS | UDP/CoAP 安全 |
| OSCORE | CoAP 对象安全 |
| SM2/SM3/SM4 | 国密签名、摘要、加密 |
| AES-GCM/CCM | 常见对称加密和认证 |
| HMAC | 消息认证 |
| Secure Element | 密钥隔离和签名认证 |

### 15.2 编码格式分层

通讯里的“编码”不是单一概念，通常分布在多个层次。

| 层次 | 解决的问题 | 常见编码/格式 |
|---|---|---|
| 字符编码 | 文本如何变成字节 | ASCII、UTF-8、GBK、UCS2、BCD |
| 业务序列化 | 业务对象如何打包 | JSON、CBOR、Protobuf、TLV、ASN.1、FlatBuffers |
| 帧封装 | 字节流如何分帧 | Length-Prefix、Delimiter、SLIP、HDLC、COBS、PPP、NMEA line |
| 字节转义 | 特殊字节如何避免冲突 | Escape、byte stuffing、bit stuffing、Base64、Hex |
| 字段编码 | 数值如何压缩 | 定点数、变长整数、bitfield、delta、dictionary、bitmap |
| 传输校验 | 检测误码 | Checksum、CRC-8/16/24/32、FCS |
| 纠错编码 | 修复误码 | Hamming、BCH、Reed-Solomon、Convolutional、LDPC、Polar |
| 交织/白化 | 抗突发错误和直流偏置 | Interleaving、Whitening、Scrambling |
| 信道编码/调制前编码 | 适配物理层 | Manchester、NRZ、4B/5B、8b/10b、DSSS、CSS |
| 安全编码 | 认证和加密 | AES-GCM、AES-CCM、SM4-CTR、HMAC、CMAC、MIC |

### 15.3 字符编码

| 编码 | 特点 | 典型协议/场景 | 注意事项 |
|---|---|---|---|
| ASCII | 7-bit 英文字符，简单 | AT、NMEA、Modbus ASCII、CLI | 不支持中文 |
| UTF-8 | Unicode 变长编码，兼容 ASCII | JSON、HTTP、MQTT payload、日志 | 中文 3 字节，长度要按字节算 |
| GBK/GB2312 | 中文本地编码 | 中文屏幕字库、老系统 | 与 UTF-8 转换要明确 |
| UCS2/UTF-16 | 2 字节或变长 Unicode | SMS PDU 中文短信 | 端序、代理对要注意 |
| BCD | 十进制数字半字节编码 | 电话号码、时间、部分表计协议 | 低/高半字节顺序常因协议不同而变化 |
| IA5 | 7-bit 字符集 | DSC、部分海事/电信协议 | 字符集受限 |
| 6-bit ASCII | 6 bit 压缩字符 | AIS payload | 不是普通 ASCII，需专门映射 |

### 15.4 文本型通讯格式

文本型格式易调试，但体积大、解析成本高。

| 格式 | 帧形态 | 典型协议 | 优点 | 缺点 |
|---|---|---|---|---|
| AT 行协议 | `AT+CMD\r`、`OK/ERROR`、URC | 蜂窝、Wi-Fi、BLE、GNSS 模组 | 人眼可读，生态成熟 | 异步 URC、透传、prompt 复杂 |
| NMEA 0183 | `$...*CS\r\n` 或 `!...*CS\r\n` | GNSS、AIS 输出 | 简单、通用 | 字段文本化，精度/效率有限 |
| JSON | `{}` 文本对象 | HTTP、MQTT、调试 API | 可读性强、工具丰富 | 开销大，裸机解析要控内存 |
| CSV | 逗号分隔 | 日志、简单遥测 | 极简 | 字段转义和兼容差 |
| INI/Key-Value | `key=value` | 配置文件、CLI | 简单 | 类型和嵌套能力弱 |
| XML | 标签文本 | 老系统、部分行业协议 | 表达力强 | 非常冗长，不适合小 MCU |

文本协议实现要点：

| 项 | 建议 |
|---|---|
| 行缓冲 | 限制最大行长，防止异常输入撑爆 buffer |
| 字段数 | 限制最大字段数量 |
| 数字转换 | 避免 `scanf` 类不可控解析，使用边界明确的 parser |
| 编码 | 明确 UTF-8/ASCII/GBK/UCS2，不要混用 |
| 转义 | JSON 字符串、CSV 引号、AT 引号都要处理 |

### 15.5 二进制序列化格式

| 格式 | 特点 | 场景 |
|---|---|---|
| JSON | 可读性强，开销大 | 调试、HTTP API |
| CBOR | 二进制对象，紧凑 | IoT |
| Protobuf | 二进制紧凑，字段可演进 | 平台协议 |

| 场景 | 推荐 |

### 15.6 TLV、定长帧与 Bitfield

TLV 是裸机通信中非常实用的编码方式。

```text
Type | Length | Value
```

常见变体：

| 变体 | 说明 |
|---|---|
| TLV | Type-Length-Value，最常见 |
| TV | 固定长度字段，省略 Length |
| LTV | Length-Type-Value，部分协议使用 |
| Nested TLV | Value 内继续嵌套 TLV |
| Bit TLV | 字段按 bit 级编码，用于极短无线包 |

定长帧适合强实时、资源极小、字段稳定的协议：

```text
Magic(1) + Ver(1) + Type(1) + Seq(1) + Payload(N) + CRC(2)
```

Bitfield 适合 LoRa、卫星短报文、AIS、Cospas-Sarsat 等低带宽链路：

| 字段 | 示例 |
|---|---|
| 坐标 | Lat 24 bit + Lon 25 bit 或更粗粒度 |
| 时间 | TOW 秒数、分钟计数、相对时间偏移 |
| 状态 | bit0=SOS、bit1=落水、bit2=低电、bit3=定位有效 |
| 编号 | 设备短 ID、事件 ID、序号 |

Bitfield 注意事项：

| 项 | 建议 |
|---|---|
| 位序 | 明确 MSB-first 还是 LSB-first |
| 字节序 | 多字节整数明确 big-endian/little-endian |
| 符号 | 明确补码、偏移量编码或无符号缩放 |
| 版本 | 必须预留 version 和 reserved bits |
| 工具 | 提供 bit layout 表和测试向量 |

### 15.7 帧封装与字节流分帧

MCU 常面对 UART/USB CDC/TCP 这种字节流，需要明确如何找帧边界。

| 分帧方式 | 形式 | 优点 | 缺点 | 典型场景 |
|---|---|---|---|---|
| 固定长度 | 每帧 N 字节 | 简单快速 | 扩展性差 | 传感器、RF 短包 |
| 长度前缀 | Len + Payload | 通用、支持二进制 | Len 错会失步 | 私有协议、Protobuf stream |
| 分隔符 | `\r\n`、0x7E 等 | 易调试 | Payload 需转义 | AT、NMEA、HDLC |
| Magic + Len | Magic + Len + Payload + CRC | 易重同步 | 略有开销 | MCU 私有协议 |
| SLIP | END/ESC 转义 | 简单 | 无内建 CRC | 串口 IP/私有协议 |
| HDLC-like | Flag 0x7E + bit/byte stuffing + FCS | 成熟可靠 | 实现稍复杂 | PPP、AX.25、部分无线 |
| COBS | Consistent Overhead Byte Stuffing | 无 0 字节，开销稳定 | 需整帧编码 | 嵌入式串口二进制 |
| CBOR sequence | 连续自描述对象 | 灵活 | parser 较复杂 | IoT 数据流 |

推荐裸机私有协议帧：

```text
Magic(2) + Version(1) + Type(1) + Seq(1) + Length(2) + Payload + CRC16(2)
```

若链路噪声大或需要快速重同步，建议 Magic 至少 2 字节，并在 header 中加入 header CRC 或 payload CRC。

### 15.8 常见协议的编码方式

| 协议/技术 | 编码/封装特点 |
|---|---|
| AT | ASCII 命令行，CR/LF 结束，URC 异步插入 |
| NMEA 0183 | ASCII，逗号字段，`*HH` XOR checksum，CR/LF 结束 |
| AIS NMEA | AIVDM/AIVDO 文本句子，payload 为 6-bit ASCII 映射 |
| SMS PDU | 十六进制文本承载二进制 PDU，号码常用 semi-octet BCD，中文常 UCS2 |
| MQTT | 二进制 fixed header + remaining length 变长编码 + topic/payload |
| MQTT-SN | 紧凑二进制消息，支持 topic id |
| CoAP | 二进制 header + token + options + payload marker |
| HTTP | 文本 header + body，body 可 JSON/CBOR/Protobuf |
| LoRaWAN | MHDR + FHDR + FPort + FRMPayload + MIC，AES-CTR/CMAC |
| BLE ATT/GATT | opcode + handle + value，运行在 L2CAP 之上 |
| Zigbee | 802.15.4 MAC + NWK + APS + ZCL，字段多为二进制 |
| Thread | 802.15.4 + 6LoWPAN + IPv6 + UDP/CoAP |
| UWB 802.15.4z | MAC frame + ranging timestamp/ST​S，二进制帧 |
| Modbus RTU | 地址 + 功能码 + 数据 + CRC16，帧间隔分帧 |
| Modbus ASCII | `:` 起始，ASCII Hex，LRC，CR/LF 结束 |
| CAN | ID + DLC + data，CAN FD 支持更长数据和 CRC |
| CANopen | CAN ID 映射 COB-ID，SDO/PDO/NMT/EMCY 对象字典 |
| J1939 | 29-bit ID，PGN/SPN 编码，TP 分包 |
| NMEA 2000 | 基于 CAN，PGN 编码船舶数据 |
| USB CDC | USB bulk endpoint 承载字节流 |
| AX.25/APRS | HDLC 风格帧，地址字段 + control + PID + info + FCS |
| DSC | 窄带 FSK，使用特定符号和海事身份/类别字段 |
| Cospas-Sarsat | 位级信标消息，BCH/CRC/扩频等信道处理 |
| ADS-B | 112-bit Mode S extended squitter，位置用 CPR 编码 |

### 15.9 数值压缩与业务字段编码

低速链路中，业务字段编码比通用 JSON 更重要。

| 数据 | 高效编码建议 |
|---|---|
| 经纬度 | `int32 x 1e7`、或按区域/精度裁剪到 24-32 bit |
| 海拔/水深 | 定点数，单位 cm/dm/m，按范围选择 bit 宽 |
| 时间 | Unix time、GNSS TOW、相对触发时间、分钟计数 |
| 电量 | 百分比 0-100、mV 定点、低电状态 bit |
| 速度/航向 | knots 或 cm/s 定点，航向 0-359 或 0-255 映射 |
| 状态 | bitmap，一字节可表达 8 个布尔状态 |
| 事件 | event_id + event_type + seq，支持幂等和去重 |
| 文本 | 词典索引、短语 ID、枚举码，避免直接传长文本 |
| 传感器 | delta 编码、scale + offset、min/max 压缩 |

示例：低速 SOS bitfield payload：

```text
Ver(3) | Type(5) | DeviceShortID(24) | EventID(16)
Lat(25) | Lon(26) | TimeOfs(16) | Battery(7)
Flags(8) | Seq(8) | CRC/MIC(...)
```

### 15.10 信道编码、纠错与交织

无线链路常常不只做业务编码，还要做信道编码。

| 编码 | 作用 | 常见场景 |
|---|---|---|
| Manchester | 时钟恢复、无直流 | 低速 RFID、红外、老式链路 |
| 4B/5B | 保证跳变密度 | 以太网/部分串行链路历史方案 |
| 8b/10b | DC balance、时钟恢复 | 高速串行链路 |
| Whitening/Scrambling | 随机化数据，减少长 0/1 | BLE、LoRa、很多 RF 芯片 |
| Interleaving | 把突发错误打散 | 卫星、海事、蜂窝、广播 |
| Hamming | 简单纠错 | 小数据、存储、低复杂度链路 |
| BCH | 多 bit 纠错 | Cospas-Sarsat、NAND、广播等 |
| Reed-Solomon | 突发错误纠正 | 存储、卫星、广播 |
| Convolutional + Viterbi | 传统无线纠错 | 卫星、蜂窝历史系统 |
| Turbo | 3G/4G 历史信道编码 | 蜂窝 |
| LDPC | 高性能纠错 | Wi-Fi、5G、卫星 |
| Polar | 5G 控制信道 | 5G NR |

### 15.11 端序、对齐与版本兼容

二进制协议必须明确以下内容：

| 项 | 建议 |
|---|---|
| 字节序 | 网络协议优先 big-endian；MCU 内部可 little-endian，但空口必须固定 |
| bit 序 | 位字段明确 bit0/bit7 先后和发送顺序 |
| 对齐 | 空口协议不要依赖 C struct padding |
| 版本 | header 中必须有 version，payload 可 TLV 扩展 |
| 保留位 | 发送填 0，接收忽略，便于未来扩展 |
| 长度 | 所有变长字段必须有最大长度 |
| 浮点 | 低端 MCU 和跨平台协议优先用定点数 |
| 字符串 | 明确编码、是否 NUL 结尾、长度单位是字节还是字符 |

不建议直接把 C 结构体 `memcpy` 到通信链路，除非明确处理了 packing、endian、alignment、版本和 CRC。

### 15.12 加密、认证与编码顺序

推荐顺序：

```text
业务对象
  -> 序列化/字段编码
  -> 压缩 optional
  -> 加密 optional
  -> 认证/MAC 或 AEAD tag
  -> 帧封装
  -> CRC/FCS optional，按链路层需求
  -> 信道编码/调制
```

注意：

| 项 | 说明 |
|---|---|
| CRC 与 MAC | CRC 检误码，MAC 防伪造，不能互相替代 |
| 先压缩后加密 | 加密后数据近似随机，无法压缩 |
| AEAD 优先 | AES-GCM/CCM、ChaCha20-Poly1305 可同时加密和认证 |
| Nonce | 不能重复，尤其 CTR/GCM/CCM |
| 明文头 | 路由字段可明文，但要纳入 AAD 认证 |

### 15.13 编码格式选择建议

## 16. 调制方式与信道编码

通讯协议最终都要落到物理层。业务编码解决“数据如何表示”，调制和信道编码解决“数据如何可靠穿过空气、线缆、水体或光路”。本章按调制、扩频、线路编码、纠错编码和典型协议对应关系整理。

### 16.1 基本概念

| 概念 | 说明 | 例子 |
|---|---|---|
| Line Coding | 比特如何映射为电平/跳变 | NRZ、Manchester、4B/5B、8b/10b |
| Modulation | 比特如何映射为载波幅度/频率/相位/脉冲 | ASK、FSK、PSK、QAM、OFDM、CSS |
| Spreading | 把窄带数据扩展到更宽频带，提高抗干扰/抗截获 | DSSS、FHSS、CSS |
| Channel Coding | 增加冗余，接收端纠错 | Hamming、BCH、RS、卷积码、LDPC、Polar |
| Interleaving | 交织打散突发错误 | 蜂窝、卫星、广播、海事链路 |
| Scrambling/Whitening | 白化数据，避免长 0/1 和谱线 | BLE、LoRa、以太网、RF 芯片 |
| Pulse Shaping | 限制频谱外泄 | GMSK Gaussian filter、RRC filter |
| Symbol Rate | 符号率，不一定等于比特率 | QPSK 每符号 2 bit，16QAM 每符号 4 bit |
| Coding Rate | 编码率，有效数据/编码后数据 | 1/2、2/3、3/4、5/6 |

### 16.2 有线链路线码

| 编码 | 特点 | 常见场景 | 注意事项 |
|---|---|---|---|
| NRZ | 0/1 直接映射电平 | UART、SPI、I2C、普通 GPIO | 长时间不跳变不利于时钟恢复 |
| NRZI | 通过是否跳变表示 bit | USB、部分磁记录/链路 | 需配合 bit stuffing |
| Manchester | 每 bit 中间必有跳变 | 以太网早期、RFID、红外 | 带宽翻倍，时钟恢复好 |
| Differential Manchester | 差分曼彻斯特 | 工业/老式链路 | 抗极性反接 |
| 4B/5B | 4 bit 转 5 bit，保证跳变 | 100BASE-TX 等历史链路 | 有 25% 开销 |
| 8b/10b | 8 bit 转 10 bit，DC balance | PCIe 早期、千兆以太网历史、SerDes | 有 25% 开销 |
| 64b/66b | 高效线路编码 | 10G Ethernet 等 | 开销低，需扰码 |
| MLT-3 | 多电平传输 | 100BASE-TX | 降低频谱需求 |
| PAM-3/PAM-4 | 多电平幅度编码 | 1000BASE-T、100G/高速 SerDes | 对信噪比和均衡要求高 |

MCU 侧常见 UART/SPI/I2C 基本是 NRZ 类数字电平，不需要固件处理线路编码；但 USB、Ethernet、PCIe、CAN 等由外设/PHY 完成线路编码和物理层细节。

### 16.3 基础无线调制

| 调制 | 全称 | 特点 | 常见场景 |
|---|---|---|---|
| OOK | On-Off Keying | 开关键控，简单低成本 | 低端遥控、315/433 MHz 简单模块、IR |
| ASK | Amplitude Shift Keying | 幅移键控 | RFID、低端 Sub-GHz |
| 2-FSK | 2-level Frequency Shift Keying | 两个频点表示 0/1 | Sub-GHz、BLE 早期物理层概念、无线表计 |
| 4-FSK | 4-level FSK | 每符号 2 bit，频谱效率高于 2-FSK | DMR、dPMR、部分工业无线 |
| GFSK | Gaussian FSK | FSK 前加高斯滤波，降低带外 | BLE、私有 2.4 GHz、Sub-GHz 芯片 |
| GMSK | Gaussian MSK | 连续相位，频谱紧凑 | GSM、AIS |
| BPSK | Binary PSK | 每符号 1 bit，鲁棒 | 卫星、DSSS、低速链路 |
| QPSK | Quadrature PSK | 每符号 2 bit，效率和鲁棒平衡 | 卫星、蜂窝、Wi-Fi |
| OQPSK | Offset QPSK | I/Q 错开，包络变化小 | IEEE 802.15.4 2.4 GHz、Cospas-Sarsat SGB 方向 |
| π/4-DQPSK | 差分 QPSK 变体 | 非相干/差分接收更方便 | TETRA、部分专业无线 |
| 8PSK | 8 相位 | 每符号 3 bit | EDGE、卫星调制 |
| 16QAM/64QAM/256QAM | 正交幅度调制 | 高频谱效率，需高 SNR | LTE、5G、Wi-Fi、Cable |
| OFDM | 正交频分复用 | 多子载波，抗多径 | Wi-Fi、LTE、5G、DVB |
| OFDMA | 多用户 OFDM | 子载波资源分配 | LTE/5G/Wi-Fi 6 |
| CSS | Chirp Spread Spectrum | 啁啾扩频，低功耗远距离 | LoRa |
| DSSS | Direct Sequence Spread Spectrum | 直接序列扩频 | GPS、802.15.4、Cospas-Sarsat、CDMA |
| FHSS | Frequency Hopping Spread Spectrum | 跳频扩频 | Bluetooth Classic、部分军警/工业无线 |
| UWB Impulse Radio | 超宽带脉冲 | 高精度测距 | IEEE 802.15.4z UWB |

### 16.5 LoRa 调制参数

LoRa 的链路预算和速率主要由 SF、BW、CR、Preamble、TX Power 决定。

| 参数 | 含义 | 影响 |
|---|---|---|
| SF | Spreading Factor，扩频因子 | SF 越大，距离越远、速率越低、空中时间越长 |
| BW | Bandwidth，带宽 | BW 越大，速率越高、灵敏度降低 |
| CR | Coding Rate，编码率 | 冗余越多，鲁棒性越高、速率越低 |
| Preamble | 前导码长度 | 影响同步能力和空中时间 |
| CRC | 包校验 | 检测误包 |
| IQ Invert | I/Q 反转 | LoRaWAN 上下行区分常用 |
| LDRO | Low Data Rate Optimization | 低速长符号时改善时钟漂移影响 |

典型取舍：

| 目标 | 参数倾向 |
|---|---|
| 最远距离 | 高 SF、低 BW、较强 CR |
| 低功耗 | 缩短 payload、降低重发、选中等 SF 减少空中时间 |
| 高容量 | 低 SF、高 BW、严格占空比控制 |
| 抗干扰 | 合理信道规划、CAD/LBT、重传、FEC |

### 16.6 BLE 调制与编码

| PHY | 调制 | 符号率/特点 | 适用场景 |
|---|---|---|---|
| LE 1M | GFSK | 1 Msym/s | 通用 BLE |
| LE 2M | GFSK | 2 Msym/s，高速短距 | OTA、日志、短距高速 |
| LE Coded S=2 | GFSK + FEC | 约 500 kbps 方向 | 较远距离 |
| LE Coded S=8 | GFSK + FEC | 约 125 kbps 方向 | 更远距离、低速 |

BLE 物理层处理包括 access address、preamble、CRC、data whitening。应用层开发通常不直接处理这些，控制器或 SoC 射频栈已经完成。

### 16.7 Wi-Fi、LTE、5G 的高阶调制

Wi-Fi、LTE、5G 通过自适应调制编码 MCS 在速率和鲁棒性之间切换。

| 条件 | MCS 倾向 |
|---|---|
| 信号强、SNR 高 | 64QAM/256QAM/1024QAM，高码率 |
| 信号弱、干扰大 | BPSK/QPSK，低码率，更强纠错 |
| 高速移动 | 更保守 MCS，更多导频/鲁棒配置 |
| 低功耗 IoT | 低带宽、低速、重复发送、深覆盖模式 |

MCU 通过普通 Wi-Fi/蜂窝模组时，一般不直接控制调制编码，只通过 AT/API 读取 RSSI、RSRP、RSRQ、SINR、MCS、速率等诊断参数。

### 16.8 海事与航空协议的调制编码

| 协议 | 调制/编码重点 | 固件关注点 |
|---|---|---|
| AIS | VHF GMSK 9600 bps，TDMA 时隙 | MMSI、位置编码、时隙/发射周期、NMEA 输出 |
| DSC | VHF 1200 bps FSK，特定 DSC 字符和呼叫格式 | 遇险类别、MMSI、位置、后续频道 |
| Cospas-Sarsat | 406 MHz 遇险信标，BCH/CRC/扩频 | 必须按标准 bit layout 和认证要求实现 |
| ADS-B | 1090 MHz PPM，112-bit Mode S | CPR 位置编码、CRC、ICAO 地址 |
| NAVTEX | NBDP/SITOR-B FSK | 接收和解码广播文本 |
| Radar SART | 雷达应答信号 | 通常由专用硬件完成 |

这类协议通常不能自由定义调制编码，必须严格按国际标准，否则接收端生态无法识别，也无法认证。

### 16.9 声学与光学调制

| 介质 | 常见调制 | 特点 |
|---|---|---|
| 水声 | FSK、PSK、OFDM、CSS、DSSS | 多径强、速率低、延迟大、多普勒明显 |
| 空气超声 | OOK、FSK、Chirp | 近距离测距/通信 |
| 红外 | OOK、PPM、脉冲距离编码 | 简单、低成本、需视距 |
| 可见光 | OOK、PPM、PWM、OFDM | 受环境光影响，适合近距或照明复用 |
| 激光 | OOK、PPM、相干调制 | 高速定向，对准要求高 |

水声通信通常需要厂商 Modem 封装复杂的同步、均衡、纠错和协议；小 MCU 直接实现完整水声物理层难度较高。

### 16.10 调制编码与 MCU 分工

| 实现位置 | MCU 需要做什么 | 例子 |
|---|---|---|
| 完整模组内部 | MCU 只发 AT/API 和 payload | 蜂窝、Wi-Fi AT 模组、BLE 模组 |
| RF 芯片硬件完成 PHY | MCU 配寄存器、FIFO、IRQ | LoRa、FSK、UWB 芯片 |
| MCU 软件生成基带 | MCU 生成 bitstream、定时、DAC/PWM | 红外、简单 OOK、低速 AFSK |
| 专用基带/Codec | MCU 配置参数和读状态 | Cospas-Sarsat、AIS、专业无线电 |
| 外部 PHY | MCU MAC/协议，PHY 做模拟/线路编码 | Ethernet、CAN、USB |

裸机 MCU 选型建议：优先选择“模组或芯片内部完成复杂 PHY”的方案，避免在 MCU 中实现高复杂度调制、同步、均衡和纠错。

## 17. 固件架构建议

### 17.1 通讯任务模型

裸机 MCU 推荐 foreground/background 模型：

```text
ISR/DMA:
  接收字节 -> ring buffer
  GPIO/IRQ -> event flag

Main loop:
  comm_device_process()
  protocol_parser_process()
  service_state_machine_process()
  low_power_manager_process()
```

### 17.2 模块抽象

| 模块 | 职责 |
|---|---|
| `comm_port_uart` | UART ring buffer、DMA、发送队列 |
| `comm_port_spi` | SPI 事务、片选、中断同步 |
| `comm_at_client` | AT 命令事务、URC 分发 |
| `comm_nmea` | NMEA 行解析、checksum、字段提取 |
| `comm_rf_chip` | RF 寄存器、FIFO、IRQ、低功耗 |
| `comm_ip` | MQTT/HTTP/CoAP/LwM2M 适配 |
| `comm_security` | TLS/DTLS/SMx/AES/HMAC/Nonce |
| `comm_codec` | JSON/CBOR/Protobuf/TLV/bitfield 编解码、端序处理 |
| `comm_framer` | Length-prefix、Delimiter、SLIP、HDLC、COBS 等分帧 |
| `comm_manager` | 多链路优先级、重试、降级、ACK |
| `comm_diag` | 日志、产测、RSSI/SNR、错误码 |

### 17.3 多链路调度原则

| 原则 | 说明 |
|---|---|
| 高优先级不阻塞 | SOS、告警、定位、关键 ACK 优先 |
| 低功耗优先 | 非关键链路默认断电或休眠 |
| 状态机清晰 | 每个模组有 power、init、attach、send、wait、error 状态 |
| 超时分级 | GNSS、蜂窝、卫星、LoRa、BLE 的超时不能共用一个常量 |
| ACK 幂等 | 事件 ID、序号、时间戳防止重复确认 |
| 协议解耦 | AT、NMEA、MQTT、LoRaWAN、Modbus 不要互相污染 |
| 可产测 | 每个链路要有自检、RSSI/SNR、环回或模拟测试方法 |

## 19. 一页总结

业界通讯模块可以按如下方式理解：

```text
广域公网：2G/3G/4G/5G/NB-IoT/LTE-M/RedCap/NTN
卫星链路：北斗短报文、天通、Iridium、Inmarsat、Globalstar
低功耗广域：LoRa P2P、LoRaWAN、Sigfox、Wi-SUN、MIOTY
短距无线：BLE、Zigbee、Thread、Wi-Fi、NFC、UWB、Sub-GHz FSK
海事救援：Cospas-Sarsat 406、AIS、DSC、NAVTEX、VHF、Radar SART
航空通信：ELT、ADS-B、UAT、VHF Airband、FLARM
专业无线：TETRA、P25、DMR、PDT、APRS、模拟 VHF/UHF
工业总线：RS-485、Modbus、CAN、CANopen、J1939、NMEA 2000、Ethernet
水下声学：水声 Modem、Pinger、Sonar、USBL/LBL
光学红外：IR、IrDA、VLC、频闪、激光通信
调试产测：SWD、JTAG、UART CLI、USB DFU、BLE DFU、NFC 标签
```

MCU 接口可以按如下方式理解：

```text
低复杂度：UART AT、I2C、GPIO
中等复杂度：SPI RF 芯片、CAN、USB Device
高复杂度：USB Host、SDIO Wi-Fi、RMII Ethernet、PCIe 5G
特殊接口：音频/I2S、ADC/DAC/PWM、PPS、RF 控制脚
```

工程选型的核心不是“协议越多越好”，而是选择能被目标场景真实接收、能通过法规认证、能被 MCU 稳定控制、能满足功耗和成本约束的链路组合。
